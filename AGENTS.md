# חוקים והנחיות טכניות (Project-Scoped Rules)

## נספח טכני: השלמת פערים למחולל הקוד (Antigravity 2.0)

מסמך זה מתרגם את הדרישות הפדגוגיות לארכיטקטורת תוכנה מוגדרת. יש להזין מפרט זה למחולל הקוד יחד עם מסמכי האפיון הקודמים כדי למנוע יצירת קוד "שבור" או חסר.

### 1. ניהול מצב (State Management) ואובייקטים אינטראקטיביים

* **ספריית Drag & Drop:** יש להנחות את הכלי להשתמש בספרייה מבוססת ומוכחת (כדוגמת `dnd-kit` או `react-beautiful-dnd`) לניהול גרירת הבלוקים הווירטואליים, כדי להבטיח ביצועים חלקים.
* **היעדר הקפצה אוטומטית (No Auto-Regrouping):** בשום שלב לא מתבצעת הקפצה אוטומטית של יחידות לעשרות.
* **קיבוץ ופריטה יזומים (Manual Grouping):** ההמרה נעשית אך ורק באופן יזום, למשל בגרירת עשרת לטור היחידות כדי לפרוט, או גרירת יחידה לטור העשרות כדי לקבץ (כאשר יש לפחות 10 יחידות).
* **מנוע סוקרטי פרואקטיבי (Proactive Socratic Friction):** החונך הסוקרטי לא מספק עובדות סגורות, אלא מייצר "חיכוך יצרני" בשאלות מנחות.
* **תצוגת ASD והתאמות ספציפיות:** המודל העיצובי הראשי הוא UDL אשר נותן מענה רחב. במידה ויהיה צורך בהתאמות נוספות (ASD), תיפתח אפשרות כזאת בעתיד שתוכל להיות מופעלת על ידי המורה באופן יזום. אין צורך ליישם isASDMode בשלב זה.
* **מקור אמת יחיד (Single Source of Truth) ומניעת כפילויות לוקאליות:** ניהול המצב של התלמידים (התקדמות, מטריצת Q, צ'אט והתראות רדאר) מנוהל אך ורק מול Firebase בזמן אמת. חל איסור מוחלט על שמירת State מתמשך (Persistence) של התלמידים ב-`localStorage` בדפדפן, כדי למנוע יצירת נתוני "רוחות רפאים" (Ghost Data).
* **ניתוב הרשאות דינמי:** מסך המורה (TeacherDashboard) ורכיבי איפוס הנתונים לא משתמשים בנתיבים קשיחים מקודדים (Hardcoded Paths כמו `teacher-1`). כל פעולות הקריאה/כתיבה/מחיקה מבוצעות על בסיס מזהה המורה הדינמי (`teacherId`).

### 2. רדאר פדגוגי שקט (Silent Radar & Trace Data)

כדי לממש ניטור סמוי ללא פגיעה בביצועים או חיווי לתלמיד:

* **טיימר רקע (Idle Timer):** הגדרת `setTimeout` של 30 שניות המופעל עם תחילת כל משימה. כל אירוע ממשק של התלמיד (`onClick`, `onDragEnd`, `onChange`) חייב להפעיל פונקציית `resetTimer`.
* **אירוע היסוס (Hesitation Event):** אם הטיימר פוקע, נשלחת בקשת רשת אסינכרונית קלה (באמצעות `WebSocket` או `fetch` רקע) שמעדכנת את סטטוס התלמיד בדשבורד המורה לחיווי "מאבק קוגניטיבי", מבלי להפריע ללקוח (ללא Spinners או חסימת מסך).
* **ניטור מחיקות/ביטולים (Sliding Window):** הגדרת חלון זמן גולש של 3 שניות שבו 3 מחיקות/ביטולים רצופים מוגדרים כ-"PASSIVE_DRIFTING". החלון מתאפס מקומית לאחר כל התראה כדי לדרוש רצף חדש, ומתאפס לחלוטין (Locked Tasks) במעבר בין משימות יחד עם איפוס טיימר ההיסוס. סך כל המחיקות למשימה נשמר בנוסף ב-`undoCount`.

### 3. מודל הנתונים (Data Schema) ואלגוריתם ה-Q-Matrix

סכמת הנתונים (JSON) קשיחה שאוספת את נתוני המיפוי (מפגש 2) לטובת הקיבוץ (Clustering) בדשבורד המורה:

```json
{
  "studentId": "string",
  "forceReload": "number",
  "conceptMastery": {
     "decimal_structure": "number",
     "number_magnitude": "number",
     "regrouping_fluency": "number",
     "procedural_fluency": "number",
     "relational_thinking": "number",
     "algebraic_reasoning": "number"
  },
  "traceData": {
     "hesitation_events": "number",
     "undo_clicks": "number",
     "interventions": {
        "timestamp_id": {
           "timestamp": "number",
           "alertType": "string",
           "responseType": "string",
           "responseText": "string"
        }
     }
  }
}
```

* **לוגיקת הקיבוץ בדשבורד:** התלמידים מקובצים לפי מושגים קוגניטיביים (Cognitive Concepts). אלגוריתם צד הלקוח/שרת יקבץ תלמידים יחד כאשר אחוז השליטה שלהם במושג מסוים נמוך מ-80% (`< 80`) ב-`conceptMastery`. לדוגמה: קבוצת תלמידים שבהם `regrouping_fluency < 80` תוגדר כקבוצה הזקוקה לחיזוק בהמרות וקיבוץ. `useStore` ו-`FirebaseSyncService` אחראים לסנכרון נתוני ה-`conceptMastery` ל-Firebase.

### 4. אינטגרציה, הזדהות ואבטחה

* **הזדהות שקופה (SSO) והתחברות:** כרגע הכניסה תהיה באמצעות רשימות נפתחות לבחירת בית ספר ומורה (Username וסיסמה קבועים מראש לתלמידים, ות.ז ושנת לידה למורים). בהמשך ייתכן ונקבע מנגנון הזדהות אוטומטית (OAuth 2.0) מול Google Classroom ללא מסך התחברות מסורבל.
* **בקרת הרשאות וניתוב (RBAC & Routing):** המערכת תנתב את המשתמש מיד לאחר ההזדהות לפי ה-Role:
* `Role: Student` מנותב ל-`/hub` (מסך בית המציג קורסים, התקדמות אישית ומשימות), ומשם נכנס למרחב העבודה ב-`/workspace`. המערכת תכלול מעטפת ניווט מלאה (App Shell) עם Sidebar ו-Topbar.
* `Role: Teacher` מנותב ל-`/dashboard` (דשבורד מתקדם המציג נתונים ורשימות תלמידים בתוך אותה מעטפת LMS).
* `Role: Admin` מנותב ל-`/admin` (מסך ניהול מערכת המציג נתוני שרתים, אבטחה, הגדרות בית ספר ומורשים).
* **איסור טיימרים צד לקוח:** יש להורות במפורש למחולל הקוד שאין לרנדר אף רכיב `<Timer />` או להציג State של זמן שחולף ב-`/workspace`.

### 5. עיצוב הממשק (UI/UX Restrictions)

* **מיקוד מסך:** הגדרת הממשק כרספונסיבי אך מוגבל בגובה (`100vh`) כדי להעלים סרגלי גלילה (Scrollbars) ככל הניתן.
* **מניעת שיוט פסיבי:** כפתור "התקדם למשימה הבאה" (`Proceed`) נשאר במצב מושבת (`disabled`) עד שהתלמיד מבצע אינטראקציה מינימלית עם מסך החקר, בהתאם לעקרון ה"חיכוך המטא-קוגניטיבי היצרני".

### 6. חוקי התנהגות (Behavioral Rules)

* **Deployment Obedience & Autonomous CI/CD (Strict Rule):** For every code update, automatically commit and push the changes to GitHub and verify that the Firebase CI/CD workflow completes successfully. If the deployment fails or an error is detected, do not stop. You must autonomously debug, fix the issue, and re-push until the update is fully deployed.
* **Sequential Task Processing:** Process all tasks strictly sequentially. Do not interrupt an ongoing task to handle a new one. If the user sends a new request, queue it and begin only after the current task is fully complete. The only exception is if the user explicitly commands you to "STOP", in which case you will halt the current task and move to the next.
* **עדכון מסמכי אפיון (Spec Updates):** כל פעם שמתבצע שינוי או חידוד להגדרות המערכת במהלך השיחה (כמו שינוי לוגיקה, הגדרות AI וכד'), יש לשאול את מנהל המערכת האם לעדכן זאת במסמכי האפיון. במידה ואישר, הסוכן יאגור 10 עדכונים כאלו, או ימתין חצי שעה מאז השיחה עליהם, ורק אז ייצר גרסה מעודכנת של מסמך האפיון וימחק את הישנה. כל זאת בזהירות כדי לא ליצור מוצר חדש או לשבור הגדרות קיימות, ותוך עדכון המשתמש.

### 7. התאמה לעקרונות העיצוב האוניברסלי ללמידה (UDL)
יש ליישם את עקרונות ה-UDL הבאים בכל בניית מודול או ממשק בתוך הפלטפורמה:
* **רכיבי תצוגה מקבילים (ייצוג):** כל מודול מתמטי מחויב למנגנון הצגה כפול לפחות. לדוגמה: ייצוג משוואה מספרית/אלגברית בצמוד לייצוג ויזואלי דינמי (כמו בדידים וירטואליים שמשתנים בזמן אמת עם המשוואה).
* **תמיכה במשוואות ונגישות (ייצוג):** חובה להשתמש בתקני רשת למתמטיקה (MathML / פונטים נגישים) המאפשרים הקראה לקוראי מסך. יש להשתמש בפלטת צבעים בעלת קונטרסט גבוה ולמנוע עומס חזותי.
* **רכיבי קלט מגוונים (ביטוי):** יש להימנע מהסתמכות בלעדית על שדות הקלדה מספרית (Inputs). חובה להשתמש במנגנוני Drag & Drop של איברים למשוואה או אפשרויות העלאת תוצר/טיוטה כיכולות ביטוי חלופיות.
* **ניווט מבוסס בחירה (מעורבות):** ארכיטקטורת המידע צריכה לתת ללומד אוטונומיה ושליטה על מסלול הלמידה שלו (לדוגמה: מסכי בחירה בין למידה מודרכת צעד-אחר-צעד או קפיצה לתרגול חקר עצמאי).
* **מנגנוני פידבק חכם (מעורבות):** במקום סימוני V/X גנריים, יש להציג משובים טקסטואליים או חזותיים בונים המציפים את הכשל הלוגי מאחורי טעות (בהתבסס על ה-Q-Matrix של התלמיד), כדי להפחית תסכול וחרדת מתמטיקה.

### 8. חוק ברזל: תקשורת תלמיד (No AI Chat)
* **איסור מוחלט על צ'אט AI:** לתלמיד אסור בשום אופן לדבר או להתכתב עם AI או עם בוטים. ה-AI פועל אך ורק מאחורי הקלעים (לעיבוד נתונים עבור המורה ולניתוב משימות), ולא מתקשר עם הילד.
* **תקשורת אנושית בלבד:** הצ'אט היחיד שפתוח לתלמיד במערכת הוא צ'אט ישיר ובלעדי מול המורה האנושי שלו. אין צ'אט בין תלמידים (No Peer-to-Peer) ואין צ'אט מול המערכת.

### 9. לקחים מתקלות: מודעות סביבתית וחקר מערכתי
* **חקר שורש לפני פלסטרים (Holistic Debugging):** במקרה של באג בממשק (למשל נתונים שלא מתעדכנים או נתונים 'מזויפים'), חובה למפות את מסלול זרימת הנתונים המלא (Data Flow) מהמשתמש ועד למסד הנתונים (Store/Firebase) ומשם לתצוגה. עליך לבדוק באופן אקטיבי את הטיפוסים ואת השדות (למשל האם משתמשים ב-uid או ב-id). אסור בשום אופן 'להדביק פלסטרים' קוסמטיים (כמו החלפת מחרוזת טקסט ספציפית בתצוגה או הזנת Mock Data) מבלי לוודא שהארכיטקטורה שמאחוריה תקינה.
* **מודעות לסביבת ההרצה (Deployment Awareness):** כאשר משתמש מדווח על באג או בודק תיקון בפרויקט שמחובר ל-CI/CD, עליך להניח שהוא מסתכל על הסביבה החיה (Production / Firebase Hosting). לכן, כל תיקון לוקאלי **חייב להיות מלווה** בפעולת Push (או בשימוש בסקיל auto_deploy). אל תבקש מהמשתמש לבדוק תיקון כל עוד הוא לא נדחף לאוויר, אלא אם וידאת שהוא מריץ סביבת פיתוח מקומית. לעולם אל תניח שתיקון לוקאלי פתר את הבעיה עבור משתמש שמסתכל באתר החי.

### 10. סנכרון אמת-קוד ותיעוד (Strict Spec Synchronization)
* **חובה עליונה ומניעת ריקבון תוכנה:** כל שינוי שמתבצע בארכיטקטורת המערכת - לרבות שינוי בנתיבי Firebase (למשל מעבר מ-`replays` ל-`telemetry_chunks`), שינוי במבנה מסד הנתונים (Data Schema), או שינוי בלוגיקת הניתוב המרכזית - חייב להיות מתועד **מיידית** במסמכי האפיון הנמצאים בתיקיית `מסמכי אפיון/` ובקובץ AGENTS.md עצמו. 
* **כלל עצירה:** אסור בשום אופן לסמן משימה כהושלמה או להציג "Walkthrough" למשתמש לפני שווידאת שכל מסמכי האפיון עודכנו ומייצגים באופן מוחלט וזהה את הקוד הקיים. כשל בעדכון המסמכים גורם לסוכנים עתידיים לכתוב קוד שגוי שמסתמך על היסטוריה לא רלוונטית.

### 11. ראייה רוחבית וניתוח השפעות (Transversal Impact Analysis)
* **בחינת השלכות לפני כל שינוי:** לפני כל שינוי נקודתי בלוגיקה, בפיזיקה או ב-UI (כמו למשל שינוי מנגנון הקפצה או הוספת רכיב), חובה עליך לבצע ניתוח רוחבי – כיצד השינוי הזה ישפיע על כל שאר חלקי המערכת (מפגשים מתקדמים, מסלול המורה, זרימת הנתונים, והחוקיות הפדגוגית).
* **הצפת משמעויות פרואקטיבית:** חובה עליך להציף למנהל המערכת את המשמעויות הרוחביות של כל הצעה לפני או מיד עם הביצוע שלה, כדי למנוע יצירת באגים רעיוניים בשלבים מתקדמים של התוכנה.

### 12. זרימת נתונים (Data Flow) וסינון הרשאות מורה
* **צ'אט בזמן אמת (Chat Sync):** מנגנון משיכת ההודעות של המורה ב-`useChatStore` חייב למשוך את כל החדרים (Rooms) ממסד הנתונים, מבלי לסנן לפי רשימת התלמידים השמורה מקומית (`useStore.getState().students`), שכן רשימה זו מכילה רק את תלמידי ההדגמה הסטטיים. סינון ההרשאות בפועל מבוצע ברמת ה-Firebase Rules, והסינון הוויזואלי מבוצע בממשק עצמו (Dashboard).
* **התראות רדאר פדגוגי (Radar Alerts):** כדי למנוע "זליגת נתונים" בין מורים שונים (Cross-Teacher Leakage), מנגנון משיכת ההתראות שואב את כל ההתראות מ-`radar_alerts`, אך חובה להפעיל סינון (`filter`) צד-לקוח כך שיוצגו אך ורק התראות של תלמידים השייכים למורה המחובר (הנמצאים ב-State המקומי `students` שעבר סינון מוקדם לפי `teacherId`). אין להציג התראות של תלמידים שאינם משויכים למורה.
* **הקלטות ולוגים (Telemetry & Replays):** כל נתוני ההקלטות (rrweb) נכתבים ונשמרים אך ורק תחת הנתיב `users/students/$studentId/telemetry_sessions`. חל איסור מוחלט על שימוש בנתיבים ישנים כמו `replays` או `telemetry_chunks`, וחל איסור על יצירת עץ `students` כפול.

### 13. חוק סקיילינג של תוכנית הלימודים (Curriculum Scaling Rule)
* **טווח המספרים לפי כיתה (Curriculum Scaling Rule):**
  * **כיתה ב' (2nd Grade):** טווח המספרים במעבדה ובניסויים מוגבל לעד 1,000.
  * **כיתה ג' (3rd Grade Adaptivity):** טווח המספרים מורחב לעד 10,000 לצורך התאמה אדפטיבית מתקדמת (כיתה ג').

### 14. חוקי זרימת המפגשים (Session Flow Rules)
* **מבנה רצף המפגשים והאישור הפדגוגי:**
  * **מפגש 1: ארגז חול (Sandbox) -** התנסות חופשית בטווח מספרים של עד 1,000.
  * **מפגש 2: אבחון ומיפוי (Diagnostic) -** סריקת רדאר ומיפוי בטווח מספרים של עד 1,000.
  * **שער אישור מורה (Teacher Approval Gate) -** לאחר מפגש 2, המערכת תינעל. המורה חייב לקרוא את האבחון מבוסס ה-AI, לבחון את הקלטות השחזור (replays/Recordings), ולבדוק את יומני הרישום (Logs) לפני מתן אישור ידני להתקדמות התלמיד.
  * **מפגשים 3-7: מפגשים אדפטיביים (Adaptive Sessions) -** ייווצרו/ייבנו אך ורק לאחר קבלת אישור המורה בשער האישור. במפגשים אלו טווח המספרים מורחב לעד 10,000.
  * **מפגש 8: אבחון וניתוח מסכם (Diagnostic Analysis) -** ניתוח אבחוני מקיף נוסף (בדומה לזה שבוצע לפני מפגש 3).

### 15. הרשאות כתיבה ואבטחה במסד הנתונים (Firebase Security Rules)
* **הרשאות כתיבה ברמת Student:** הוגדר חוק `.write` מפורש בנתיב `users/students/$studentId` המאפשר לתלמיד עצמו, למורה שלו או למנהל המערכת לכתוב ישירות לתוך עץ התלמיד.
  ```json
  ".write": "auth != null && (auth.token.email === $studentId + '@mathmaticore.local' || root.child('users/teachers/' + auth.token.email.replace('teacher_', '').replace('@mathmaticore.local', '')).exists() || auth.token.email === 'admin@mathmaticore.local')"
  ```
  הרשאה זו מחלחלת (cascades) ומאפשרת כתיבה לענפי הילדים:
  - `conceptMastery`: אחוזי שליטה למיפוי קוגניטיבי.
  - `interaction_logs`: הקלטות telemetry ויומני פעילות.
  - `routeRecommendation` ו-`routeStatus`: המלצות הנתב ומצב המסלול.
* **מחיקת חדר צ'אט:** תחת `chat_messages/$roomId` הוגדרה הרשאת כתיבה מיוחדת המאפשרת מחיקה מלאה של החדר (כאשר `newData.exists() === false`).

### 16. מנגנון טיפול בשגיאות ומניעת כישלונות שקטים (Error Boundaries & UI Alerts)
* **זרימת הודעות צ'אט (useChatStore):**
  - כשל בכתיבה ל-Firebase (`firebaseSet`) תחת `sendMessage` או `sendImageMessage` נתפס ונרשם בקונסול, ומפעיל התראה למשתמש: `"שגיאה בשליחת ההודעה. אנא נסה שוב."`.
* **תהליך הרפלקציה (ReflectionScreen):**
  - בעת שמירת הרפלקציה או קריאה ל-Socratic Engine להכנת משימות ה-AI, במידה ואחת הפעולות נכשלת:
    1. מוצגת התראה: `"אירעה שגיאה בשמירת הרפלקציה והכנת המשימות הבאות. אנא נסה שוב."`.
    2. ה-State של הכפתור `done` מוחזר ל-`false` כדי לאפשר למשתמש ללחוץ שוב ולנסות שנית.
    3. מתבצע `return` כדי למנוע ניווט אוטומטי למסך הבית (`/hub`) עם נתונים חסרים.
* **אישור ודחיית משימות מורה (TeacherDashboard):**
  - אישור משימות (`approveTasks`) ודחייתן (`rejectTasks`) עטופים בבלוק `try/catch` המציג התראה למורה במקרה של כשל תקשורת או הרשאות ב-Firebase.

### 17. הפרדת רשויות הרמטית במסמכים (Strict Separation of Concerns in Documentation)
* **לכל מסמך אפיון יש מטרה אחת בלבד (Single Responsibility Principle):** חל איסור מוחלט על "זיהום" מסמכים במידע שלא שייך אליהם. כאשר יש הנחיה לעדכן מסמכים, חובה לשמור על הגבולות הבאים:
  * `מסמך מאסטר פדגוגי`: חזון ומחקר בלבד. אין להכניס שום מידע ארכיטקטוני או קוד!
  * `אפיון רצף פעילויות`: הפדגוגיה, המשימות והמטריצות הקוגניטיביות נטו (Q-Matrix, רצף שיעורים 1-8). אין להכניס הגדרות Firebase, או מודלים של נתונים!
  * `ארכיטקטורת המידע`: עדכוני שרת, Firebase Rules, Data Schema וניהול State.
  * `מסמך אפיון מפורט`: UI/UX, מנעולי התקדמות (Progression Locks) וממשקי משתמש (Teacher Co-Pilot).
* **אזהרת Spec Updater:** כאשר משתמשים בסקיל של עדכון המסמכים, חובה לוודא שלא מבצעים Copy-Paste עיוור של עדכוני ארכיטקטורה לתוך המסמך הפדגוגי.
* **עדכון תאריכים מבוקר:** חל איסור לעדכן את השעה והתאריך בראש מסמך אלא אם כן בוצע בו שינוי תוכן משמעותי ואמיתי. אין לדרוס תאריכים במסמכים שלא השתנו.
