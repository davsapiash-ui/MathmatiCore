# חוקים והנחיות טכניות (Project-Scoped Rules)

## נספח טכני: השלמת פערים למחולל הקוד (Antigravity 2.0)

מסמך זה מתרגם את הדרישות הפדגוגיות לארכיטקטורת תוכנה מוגדרת. יש להזין מפרט זה למחולל הקוד יחד עם מסמכי האפיון הקודמים כדי למנוע יצירת קוד "שבור" או חסר.

### 1. ניהול מצב (State Management) ואובייקטים אינטראקטיביים

* **ספריית Drag & Drop:** יש להנחות את הכלי להשתמש בספרייה מבוססת ומוכחת (כדוגמת `dnd-kit` או `react-beautiful-dnd`) לניהול גרירת הבלוקים הווירטואליים, כדי להבטיח ביצועים חלקים.
* **מכונת מצבים להמרה (Regrouping State):** יש להגדיר לוגיקה המאזינה למיקום ולכמות הבלוקים ב"טבלת הטורים".
* **הקפצה (חיבור):** כאשר המערך בטור היחידות מונה 10 פריטים (`array.length === 10`), ה-State של אותן יחידות נמחק, ובמקומו נוצר אובייקט "עשרת" בטור העשרות.
* **פריטה (חיסור):** גרירת אובייקט "עשרת" לטור היחידות מעדכנת את ה-State כך שהעשרת נמחקת ונוצרים 10 אובייקטים מסוג "יחידה".
* **תצוגת ASD סמויה וגלויה:** ניהול משתנה גלובלי `isASDMode (boolean)`. כאשר הוא פעיל, יש להפעיל פילטרים ב-CSS (למשל `opacity: 0.3` לטורים שאינם במוקד הפעילות) ולהשבית אנימציות מעבר מורכבות למניעת הצפה חושית.

### 2. רדאר פדגוגי שקט (Silent Radar & Trace Data)

כדי לממש ניטור סמוי ללא פגיעה בביצועים או חיווי לתלמיד:

* **טיימר רקע (Idle Timer):** הגדרת `setTimeout` של 15 שניות המופעל עם תחילת כל משימה. כל אירוע ממשק של התלמיד (`onClick`, `onDragEnd`, `onChange`) חייב להפעיל פונקציית `resetTimer`.
* **אירוע היסוס (Hesitation Event):** אם הטיימר פוקע, נשלחת בקשת רשת אסינכרונית קלה (באמצעות `WebSocket` או `fetch` רקע) שמעדכנת את סטטוס התלמיד בדשבורד המורה לחיווי "מאבק קוגניטיבי", מבלי להפריע ללקוח (ללא Spinners או חסימת מסך).
* **ניטור מחיקות/ביטולים:** הגדרת משתנה `undoCount (number)` הנשמר ב-State של המשימה ומתעדכן בכל לחיצה על כפתור ביטול הפעולה.

### 3. מודל הנתונים (Data Schema) ואלגוריתם ה-Q-Matrix

יש להגדיר סכמת נתונים (JSON) קשיחה שאוספת את נתוני המיפוי (מפגש 2) לטובת הקיבוץ (Clustering) בדשבורד המורה:

```json
{
  "studentId": "string",
  "qMatrixResults": {
     "task1_zero_placeholder": "boolean",
     "task2_estimation_error_margin": "number",
     "task3_flexible_regrouping": "boolean",
     "task4_basic_addition_fluency": "boolean"
  },
  "traceData": {
     "hesitation_events": "number",
     "undo_clicks": "number"
  }
}
```

* **לוגיקת הקיבוץ בדשבורד:** אלגוריתם צד השרת יקבץ תלמידים לפי ערכי שקר (`false`) ב-`qMatrixResults`. לדוגמה: קבוצת תלמידים שבהם `task4_basic_addition_fluency === false` תוגדר כקבוצה הזקוקה לחיזוק עובדות יסוד (כיתה א').

### 4. אינטגרציה, הזדהות ואבטחה

* **הזדהות שקופה (SSO):** הטמעת Auth מבוסס Google Workspace. הכניסה מה-Google Classroom תמשוך אוטומטית את מזהה התלמיד (`OAuth 2.0`) ללא מסך התחברות מסורבל.
* **בקרת הרשאות וניתוב (RBAC & Routing):** המערכת תנתב את המשתמש מיד לאחר ההזדהות לפי ה-Role:
* `Role: Student` מנותב ישירות ל-`/workspace` (מסלול לינארי, ללא תפריטי ניווט עליונים/צדדיים).
* `Role: Teacher` מנותב ל-`/dashboard`.
* **איסור טיימרים צד לקוח:** יש להורות במפורש למחולל הקוד שאין לרנדר אף רכיב `<Timer />` או להציג State של זמן שחולף ב-`/workspace`.

### 5. עיצוב הממשק (UI/UX Restrictions)

* **מיקוד מסך:** הגדרת הממשק כרספונסיבי אך מוגבל בגובה (`100vh`) כדי להעלים סרגלי גלילה (Scrollbars) ככל הניתן.
* **מניעת שיוט פסיבי:** כפתור "התקדם למשימה הבאה" (`Proceed`) נשאר במצב מושבת (`disabled`) עד שהתלמיד מבצע אינטראקציה מינימלית עם מסך החקר, בהתאם לעקרון ה"חיכוך המטא-קוגניטיבי היצרני".

### 6. חוקי התנהגות (Behavioral Rules)

* **Deployment Obedience:** כאשר המשתמש מורה בפירוש לעדכן את גיטהאב/האתר הפעיל (לדוגמה: "I want you to commit and push the code to github and make sure that the cicd to firebase runs"), חובה על הסוכן לציית באופן מיידי ולבצע את פעולות ה-Commit וה-Push ללא שאלות או צורך באישור נוסף.
